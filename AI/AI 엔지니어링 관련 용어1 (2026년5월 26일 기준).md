# AI 개발 핵심 용어 사전 (2026년 5월 기준)

> 현업에서 실제로 쓰이는 용어 기준으로 정리. 단순 정의보다 **어디서 어떻게 쓰이는지** 중심으로 서술.
> 🆕 = 2026년 신규 추가 / ⚡ = 내용 업데이트

---

## 1. 프롬프트 엔지니어링 (Prompt Engineering)

**한 줄 정의**: LLM이 원하는 결과를 출력하도록 입력(프롬프트)을 설계하는 기술.

**핵심 기법**

- **Zero-shot**: 예시 없이 지시만으로 수행
- **Few-shot**: 2~5개 예시를 함께 제공해 패턴 학습 유도
- **Chain-of-Thought (CoT)**: "단계적으로 생각해봐"처럼 추론 과정을 명시적으로 요구
- **System Prompt 설계**: 역할, 말투, 제약 조건을 사전에 정의
- **Negative Prompting**: 하지 말아야 할 것을 명시

**현재 위상**: 모델 성능이 높아지면서 단독으로는 비중이 줄었음. 그러나 에이전트 시스템 설계와 하네스 구성에서는 여전히 핵심 기반 기술.

---

## 2. 바이브 코딩 (Vibe Coding) 🆕

**한 줄 정의**: AI에게 원하는 것을 자연어로 설명하고, 코드 구현은 AI에게 완전히 맡기는 개발 방식. 2025년 2월 Andrej Karpathy가 처음 제안한 용어.

**특징**

- 개발자가 코드를 직접 작성하지 않고 AI 출력을 검토·승인
- 프로토타이핑과 일회성 스크립트에 특히 효과적
- "코드가 존재한다는 사실조차 잊어버리는" 수준의 추상화

**한계**: 프로덕션 배포 시 유지보수·보안 문제가 빠르게 부각됨. 이를 보완하기 위해 **에이전틱 엔지니어링**으로 진화.

```
Karpathy는 2026년 2월 1년 회고에서 "바이브 코딩은 설렘을 포착한 용어였지만, 다음 단계는 에이전틱 엔지니어링"이라고 직접 언급.
```

---

## 3. 에이전틱 엔지니어링 (Agentic Engineering) ⚡

**한 줄 정의**: AI 에이전트가 코드를 계획·작성·테스트·배포하는 시스템을 **구조화된 인간 감독 하에** 설계하는 분야. 바이브 코딩의 성숙한 형태. (Karpathy, 2026)

**바이브 코딩과의 차이**
| | 바이브 코딩 | 에이전틱 엔지니어링 |
|--|--|--|
| 철학 | AI 출력을 신뢰하고 흘러가기 | AI를 감독하고 검증하기 |
| 적합 환경 | 프로토타입, 주말 프로젝트 | 프로덕션, 팀 단위 개발 |
| 인간 역할 | 결과 수용자 | 문제 정의자 + 품질 판단자 |

**핵심 원칙**

- **에이전틱**: 99%의 코드는 에이전트가 작성, 개발자는 오케스트레이션
- **엔지니어링**: 전문성, 구조, 검증이 수반되는 규율 있는 방법론

**구성 요소**
| 요소 | 설명 |
|------|------|
| Planner | 목표를 하위 태스크로 분해 |
| Tool Use | 웹 검색, 코드 실행, API 호출 등 |
| Memory | 단기(컨텍스트) / 장기(벡터 DB) 기억 |
| Executor | 실제 액션 수행 |
| Evaluator | 결과 검증 후 재시도 판단 |

**대표 패턴**

- **ReAct** (Reasoning + Acting): 생각하고 → 행동하고 → 관찰하는 루프
- **Plan-and-Execute**: 전체 계획 수립 후 순차 실행
- **Multi-Agent**: 역할이 다른 에이전트들이 협업

**주요 프레임워크**: LangGraph, AutoGen, CrewAI, smolagents

## 4. 하네스 엔지니어링 (Harness Engineering) 🆕

**한 줄 정의**: AI 코딩 에이전트가 대규모 환경에서 안정적으로 작동할 수 있도록 **환경, 제약 조건, 피드백 루프**를 설계하는 분야.

**왜 등장했나**: 에이전트가 코드를 대량 생성할수록 품질·보안 관리가 병목이 됨. OpenAI는 이 방식으로 사람이 직접 작성한 코드 없이 100만 줄 이상의 코드를 구축한 사례가 있음.

**핵심 구성**

- **환경(Environment)**: 에이전트가 작동할 샌드박스, 도구, 리소스 정의
- **제약(Constraints)**: 에이전트가 할 수 있는 것/없는 것의 명확한 경계
- **피드백 루프**: 테스트 자동화, 린팅, CI/CD로 에이전트 출력 즉시 검증
- **품질 게이트(Quality Gates)**: 보안 스캐닝, 코드 리뷰, 스타일 체크를 파이프라인에 내장

**보안 주의**: 주당 1,000개의 PR을 생성하는 에이전트의 취약점 발생률이 1%라면 매주 10개의 새로운 취약점이 생김. 보안을 나중에 덧붙이지 말고 하네스 자체에 구축해야 함.

```
에이전틱 엔지니어링이 "무엇을 만들지"를 다룬다면, 하네스 엔지니어링은 "어떤 환경에서, 어떤 제약으로 만들지"를 다룸.
```

## 5. RAG (Retrieval-Augmented Generation) ⚡

**한 줄 정의**: LLM의 답변을 생성할 때 외부 문서를 실시간으로 검색해 컨텍스트로 주입하는 기법.

**기본 파이프라인**

```
질문 → 임베딩 → 벡터 DB 검색 → 관련 문서 추출 → LLM에 함께 전달 → 답변 생성
```

**고급 RAG 기법 (2025~2026)**

- **Hybrid Search**: 키워드(BM25) + 시맨틱 검색 혼합
- **Re-ranking**: 검색 결과를 Cross-Encoder로 재정렬
- **GraphRAG**: 문서를 지식 그래프로 구조화해 검색
- **Agentic RAG**: 에이전트가 검색 전략을 스스로 결정하고 다중 검색을 조율. 2026년 기준 가장 주목받는 아키텍처
- **Contextual Retrieval**: Anthropic이 제안, 청크에 컨텍스트를 추가해 검색 정확도 향상

**핵심 지표**: Recall@K, MRR, RAGAS 점수

## 6. 파인튜닝 (Fine-tuning) vs PEFT

**한 줄 정의**: 사전학습된 모델을 특정 태스크/도메인 데이터로 추가 학습시키는 것.

**종류**
| 방식 | 설명 | 비용 |
|------|------|------|
| Full Fine-tuning | 전체 파라미터 업데이트 | 매우 높음 |
| LoRA | 저랭크 행렬만 학습 | 낮음 |
| QLoRA | 양자화 + LoRA | 매우 낮음 |
| Prefix Tuning | 입력에 학습 가능한 토큰 추가 | 낮음 |

**언제 쓰나**: RAG로 해결 안 되는 **말투/형식/도메인 특화 지식** 학습 시. 단, 데이터 수집·학습 비용이 크므로 RAG를 먼저 시도하는 것이 일반적.

## 7. 에이전트 오케스트레이션 (Agent Orchestration) ⚡

**한 줄 정의**: 여러 에이전트 또는 LLM 호출을 조율하는 상위 레이어 설계.

**패턴**

- **Supervisor 패턴**: 마스터 에이전트가 하위 에이전트에게 태스크 분배
- **Swarm 패턴**: 에이전트들이 자율적으로 서로에게 핸드오프
- **Sequential Chain**: A → B → C 순서로 결과를 넘김
- **Parallel Fan-out**: 여러 에이전트 동시 실행 후 결과 병합

**2026년 트렌드**: Gartner 기준 멀티 에이전트 시스템 도입 문의가 2024 Q1 대비 2025 Q2에 1,445% 급증. 단일 에이전트 → 전문화된 에이전트 팀으로의 전환이 가속 중.

**현업 포인트**: 오케스트레이션이 복잡할수록 디버깅이 어려워짐. 관찰 가능성(Observability) 설계가 필수.

## 8. 컨텍스트 엔지니어링 (Context Engineering) ⚡

**한 줄 정의**: LLM의 컨텍스트 창(Context Window)에 **무엇을, 어떤 순서로, 얼마나** 넣을지 설계하는 기술. 2025~2026년 가장 주목받는 용어 중 하나.

**왜 중요한가**

- 주요 모델들이 100K~수백만 토큰 컨텍스트를 지원
- 하지만 컨텍스트가 길어질수록 **중간 정보를 잃는 현상(Lost in the Middle)** 발생
- 비용도 토큰 수에 비례

**핵심 전략**

- 중요한 정보는 **앞과 끝**에 배치
- 불필요한 내용은 RAG나 요약으로 압축
- 대화 히스토리를 선택적으로 포함
- 구조화된 형식(XML, JSON)으로 명확하게 구분

```
프롬프트 엔지니어링의 진화된 형태. 에이전트 시스템에서는 컨텍스트 설계가 곧 시스템 설계.
```

## 9. 함수 호출 / 도구 사용 (Function Calling / Tool Use)

**한 줄 정의**: LLM이 외부 함수나 API를 호출할 수 있도록 인터페이스를 정의하는 기능.

**작동 방식**

```
사용자 질문 → LLM이 함수 호출 필요성 판단
→ 함수명 + 인자를 JSON으로 출력
→ 실제 함수 실행 (개발자 코드)
→ 결과를 LLM에 다시 전달
→ 최종 답변 생성
```

**활용**: 날씨 조회, DB 쿼리, 계산기, 외부 API 연동 등. 에이전틱 시스템의 핵심 메커니즘.

**주의**: 함수 스키마 설계가 곧 프롬프트 엔지니어링. 설명이 불명확하면 LLM이 잘못된 함수를 호출함.

## 10. MCP (Model Context Protocol) ⚡

**한 줄 정의**: **AI 모델과 외부 도구/데이터 소스를 연결하는 표준 프로토콜**. Anthropic이 2024년 공개 후 2025년 12월 Linux Foundation에 기증.

**MCP의 해결책**

- 표준화된 서버-클라이언트 구조
- MCP 서버를 한번 만들면 Claude, Cursor, VS Code, OpenAI 등 어디서든 재사용
- Resources(데이터), Tools(함수), Prompts(템플릿) 세 가지 인터페이스 제공

**현황 (2026년 초)**: 월간 SDK 다운로드 9,700만 건 돌파. OpenAI, Google, Microsoft, Amazon 등 모든 주요 AI 제공사가 공식 채택. GitHub, Slack, Google Drive, Figma 등 수천 개의 MCP 서버가 공개됨.

```
MCP = 에이전트의 "손". A2A와 함께 에이전트 프로토콜 스택의 핵심 레이어.
```
