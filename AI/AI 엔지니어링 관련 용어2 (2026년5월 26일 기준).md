## 11. A2A (Agent-to-Agent Protocol) 🆕

**한 줄 정의**: **AI 에이전트끼리 서로 발견하고 소통하고 협업**할 수 있도록 표준화한 프로토콜. Google이 2025년 4월 제안, 같은 해 Linux Foundation에 기증.

**MCP와의 차이**
| | MCP | A2A |
|--|--|--|
| 역할 | 에이전트 ↔ 도구/데이터 연결 | 에이전트 ↔ 에이전트 연결 |
| 비유 | 에이전트의 "손" | 에이전트의 "언어" |
| 주도 | Anthropic | Google |

**현황**: IBM의 ACP(Agent Communication Protocol)가 2025년 8월 A2A에 통합. 2025년 12월 OpenAI, Anthropic, Google, Microsoft, AWS, Block이 공동 창립한 **Agentic AI Foundation(AAIF, Linux Foundation 산하)** 아래 MCP와 A2A가 함께 관리됨.

**3계층 에이전트 프로토콜 스택**

```
WebMCP  — 웹 접근 레이어
  A2A   — 에이전트 간 통신 레이어
  MCP   — 도구/데이터 연결 레이어
```

## 12. 평가 / 이발스 (Evaluation / Evals) ⚡

**한 줄 정의**: LLM 또는 에이전트 시스템의 성능을 측정하는 체계적인 방법론.

**평가 유형**
| 유형 | 방법 |
|------|------|
| 자동 평가 | 정답 비교, LLM-as-Judge |
| 인간 평가 | A/B 비교, 루브릭 채점 |
| 태스크별 평가 | RAGAS(RAG), HumanEval(코드), MT-Bench(대화) |

**LLM-as-Judge**: 다른 LLM으로 출력을 채점. 빠르지만 편향 주의.

**2026년 현황**: SWE-bench Verified(소프트웨어 엔지니어링 벤치마크)에서 상위 모델들이 거의 100% 달성. 기존 벤치마크가 포화 상태에 이르러 새로운 평가 기준 개발이 업계 과제로 부상.

**현업 포인트**: 2026년에는 Evals 파이프라인 구축이 MLOps의 필수 요소로 정착. "느낌"으로 품질 관리하는 방식은 프로덕션에서 통하지 않음.

## 13. 가드레일 / 안전장치 (Guardrails)

**한 줄 정의**: LLM 출력이 유해하거나 부적절하거나 형식에 맞지 않을 때 이를 차단·수정하는 레이어.

**종류**

- **Input Guardrails**: 악의적 프롬프트 인젝션 탐지
- **Output Guardrails**: 욕설, 개인정보, 환각(Hallucination) 필터링
- **Format Guardrails**: 출력이 지정한 JSON 스키마를 따르는지 검증

**주요 도구**: Guardrails AI, NVIDIA NeMo Guardrails, Llama Guard

## 14. 에이전트 메모리 (Agent Memory)

**한 줄 정의**: 에이전트가 정보를 저장하고 이후 작업에 활용하는 방식.

**4가지 메모리 유형**
| 유형 | 설명 | 예시 |
|------|------|------|
| Sensory | 현재 입력/컨텍스트 | 지금 대화 내용 |
| In-context (단기) | 컨텍스트 창 내 저장 | 세션 중 대화 히스토리 |
| External (장기) | 벡터 DB, KV 스토어 | 사용자 선호도, 과거 작업 |
| Procedural | 행동 패턴 학습 | 파인튜닝된 모델 가중치 |

---

## 15. 구조화된 출력 (Structured Output)

**한 줄 정의**: LLM이 자유 텍스트 대신 JSON, XML 등 정해진 스키마로 출력하도록 강제하는 기법.

**방법**

- **Constrained Decoding**: 문법적으로 유효한 JSON만 생성하도록 토큰 제한 (Outlines, Guidance 라이브러리)
- **API 레벨 지원**: OpenAI `response_format`, Anthropic의 Tool Use 활용
- **프롬프트 기반**: 스키마를 프롬프트에 명시 (신뢰도 낮음)

## 16. 멀티모달 (Multimodal) ⚡

**한 줄 정의**: 텍스트 외에 이미지, 오디오, 비디오, 코드 등 여러 형태의 데이터를 처리할 수 있는 모델 또는 시스템.

**현황 (2026)**

- **Physical AI**: 로봇에 멀티모달 AI를 탑재해 실세계에서 작동하는 시스템. CES 2026에서 다수 회사가 휴머노이드 로봇 시연. Jensen Huang: "로봇의 ChatGPT 모멘트가 왔다"
- **World Model**: 물리 법칙을 이해하고 시뮬레이션하는 모델. Google DeepMind Genie 3, NVIDIA Cosmos Predict 2.5 등
- OpenAI Sora 2, Google Veo 3.1 등 영상 생성 도구가 실용 수준에 도달

## 17. 추론 모델 (Reasoning Model) ⚡

**한 줄 정의**: 답변 전 내부적으로 **긴 추론 과정(Chain-of-Thought)을 먼저 수행**하도록 훈련된 모델.

**대표 모델 (2026년 기준)**

- OpenAI o3, o4-mini
- Claude Sonnet 4.6 / Opus 4.6 (Extended Thinking)
- Gemini 2.5 Flash/Pro
- DeepSeek-R1 (오픈소스)

**특징**: 수학, 코딩, 논리 문제에서 기존 모델 대비 큰 성능 향상. 단, 응답 속도가 느리고 비용이 높음.

**활용 포인트**: 복잡한 단계적 추론이 필요한 태스크에만 선택적으로 적용. 모든 태스크에 추론 모델을 쓰는 건 낭비.

## 18. 오픈 웨이트 모델 (Open-weight Model) 🆕

**한 줄 정의**: 모델의 가중치(파라미터)를 공개해 누구나 다운로드·배포·수정할 수 있는 모델. "오픈소스 모델"과 혼용되지만 엄밀히는 다름(학습 데이터·코드 공개 여부에 따라 구분).

**현황 (2026)**

- **Meta LLaMA 4**: 강력한 멀티모달 오픈 웨이트 기반 모델
- **DeepSeek-R1**: 추론 특화, 중국 오픈 생태계 대표주자
- **Alibaba Qwen3**: 다국어 및 코딩에 강점
- **OpenAI gpt-oss**: 2025년 8월 GPT-2 이후 첫 오픈 웨이트 공개 (120B, 20B, Apache 2.0)
- 상위 클로즈드 모델과의 성능 격차가 크게 줄어 실용성 급상승

**활용 포인트**: 데이터 프라이버시, 비용, 커스터마이징이 중요한 경우 오픈 웨이트 모델이 유리.

## 19. AI 거버넌스 / 규제 (AI Governance) 🆕

**한 줄 정의**: AI 시스템의 개발·배포·사용에 관한 규칙, 정책, 법적 프레임워크.

**주요 동향 (2026)**

- **EU AI Act**: 일반 목적 모델(GPAI)에 대한 의무 사항이 2025년 8월부터 적용 시작
- **NIST AI RMF**: 미국 기준 AI 위험 관리 프레임워크. 엔터프라이즈 AI 배포의 참조 표준
- **AI 감사(Audit)**: 모델 편향, 설명가능성, 데이터 출처 검증이 규제 대상으로 부상

**현업 포인트**: 규제를 "속도 저하 요인"이 아닌 "신뢰 구축 요소"로 접근하는 조직이 오히려 빠르게 스케일업하는 추세.

## 20. 비용 최적화 관련 용어 ⚡

### 캐싱 (Prompt Caching)

반복되는 시스템 프롬프트나 문서를 캐시해 재사용 → 비용 절감. Anthropic, OpenAI 모두 지원.

### 모델 라우팅 (Model Routing)

쉬운 태스크는 작은 모델(Haiku, GPT-4o-mini), 복잡한 태스크는 큰 모델로 자동 분기.

### 배치 처리 (Batch API)

비실시간 태스크를 묶어서 처리 → 최대 50% 비용 절감. (Anthropic Message Batches API)

### 엣지 AI (Edge AI) 🆕

클라우드가 아닌 기기(스마트폰, IoT, 서버 온프레미스) 위에서 AI 모델을 직접 실행. 지연시간 감소·개인정보 보호에 유리. 2026년 하이프에서 실용화 단계로 진입 중.

### Agent FinOps 🆕

에이전트가 자율적으로 API를 호출하고 리소스를 소비하는 환경에서 비용을 추적·예측·최적화하는 방법론. 에이전트가 많아질수록 비용 관리가 별도 역할로 분화됨.

## 21. 인지 부채 (Cognitive Debt) 🆕

**한 줄 정의**: AI가 생성한 코드와 인터랙션을 제대로 관리하지 않아 누적되는 **컨텍스트 손실, 에이전트 행동 불예측성, 신뢰 저하** 비용. 2026년 기술 부채(Technical Debt)의 다음 버전으로 부상한 개념.

**왜 등장했나**: 2025년에는 기술 부채가 엔지니어링 팀의 주요 부담이었다면, 2026년에는 잘못 관리된 AI 인터랙션의 누적 비용이 새로운 주요 위협으로 대두.

**해결 방향**: 에이전트 행동 로깅, 컨텍스트 보존 전략, 에이전트 출력 검토 프로세스 정착.

## 용어 간 관계 요약 (2026 업데이트)

```
프롬프트 엔지니어링
    └── 컨텍스트 엔지니어링 (진화형)

바이브 코딩 (2025)
    └── 에이전틱 엔지니어링 (2026, 성숙한 형태)
            ├── 하네스 엔지니어링 (환경/제약 설계)
            ├── RAG / Agentic RAG (지식 주입)
            ├── 함수 호출 + MCP + A2A (도구/에이전트 연동)
            │       └── [MCP: 에이전트↔도구] [A2A: 에이전트↔에이전트]
            ├── 에이전트 메모리 (상태 관리)
            ├── 오케스트레이션 (다중 에이전트)
            ├── 가드레일 + Evals (품질/안전 관리)
            └── Agent FinOps + 인지 부채 관리 (비용/부채 관리)

멀티모달 / 추론 모델 / 오픈 웨이트 모델
    └── 위 시스템의 기반 모델 레이어
```
